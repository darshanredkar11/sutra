The $2T Problem Sutra Solves

Software failure is the largest operational risk most companies don't measure. Existing tools catch syntax and style — they don't predict production risk. Sutra bridges the gap between static analysis and production reliability.

The Problem

  • Linters check style, not risk. They catch formatting but can't predict runtime failures.
  • Code quality platforms compute scores that don't correlate with incident frequency.
  • APM tools (DataDog, Sentry) detect failures after they reach production — too late.
  • ML models require labeled defect data that most teams don't have.
  • Security scanners (CodeQL, Semgrep) find vulnerabilities but miss architectural and process risk.

Why Sutra

  • 7 engines, one pipeline — complexity + dependency + RSE + process + ML + LLM + HITL, fused into a single risk score
  • Zero external deps — runs on any system with Rust and a filesystem. No databases, no queues, no SaaS needed
  • Deterministic by default — same repo + commit = identical results. ML and LLM are optional enrichment
  • Open core — all engines are Apache 2.0. Enterprise features (SSO, audit, hosted) are additive
  • CI-native — SARIF 2.1.0 output integrates with GitHub Advanced Security, GitLab, and any CI pipeline

How Each Engine Works

Every engine exists to answer a specific question about production risk. Together they form a holistic picture no single tool can provide.

mgtg — Complexity Engine sutra-mgtg

Why: Code complexity is the oldest and most validated predictor of defects. McCabe's 1976 study established that functions with cyclomatic complexity > 10 are statistically more likely to contain faults. Cognitive complexity extends this by weighting nesting depth — an if inside a for inside a while costs more than three sequential ifs.

Logic

M = E − N + 2P — cyclomatic number counts linearly independent paths through the CFG. Cognitive complexity adds weighted penalties for nesting depth and logical operators.

Output

Per-file metrics: cyclomatic_max, cognitive_max, nesting_max, total_functions. Findings for functions exceeding thresholds.

dependency — Graph Analysis Engine sutra-dependency

Why: Circular dependencies couple modules that should be independent. High fan-in modules are fragile foundations; high fan-out modules are brittle hubs. Architecture layers enforce explicit dependency direction (presentation → application → domain).

Logic

Builds directed graph G = (V, E). Tarjan's SCC (O(V+E)) partitions nodes into strongly connected components. SCCs with >1 node = circular dependencies. Fan-in = indegree, fan-out = outdegree.

Output

DependencyGraph with nodes, edges, cycles, fan-in/out maps, architecture violations. Findings for each cycle and layer violation.

process — Change Mining Engine sutra-process

Why: The best predictor of future failure is past change patterns. Files with frequent edits, many authors, or evenly distributed change (high entropy) are statistically riskier. This is the foundation of JIT defect prediction [Kamei et al., 2013].

Logic

Hassan entropy H(f) = −Σ pᵢ · log₂(pᵢ). 14 JIT features per file from git history. Co-change matrix surface hidden coupling between files in different packages.

Output

Per-file FeatureMap (HashMap<String, f64>) with all 14 JIT features persisted to SQLite. Findings for high entropy, churn, and coupling.

rse — Runtime Survivability Engine sutra-rse

Why: Complexity and dependencies only tell you if code is well-structured — they don't tell you if it will survive production traffic. RSE models queueing theory (M/M/1, Little's Law), memory pressure, CPU saturation, GC pressure, and thread starvation for every endpoint. Predicts operational failure before a single request is served.

Logic

Weight estimation from JSON schema (raw bytes × runtime expansion). Queueing via M/M/1 utilization ρ = λ/µ, active requests N = ρ/(1−ρ), response time R = (1/µ)/(1−ρ). CPU/memory/GC/thread/latency risk from complexity profile. Final survivability = 1 − max(component risks).

Output

SurvivabilityScore [0,1] per endpoint. 7 finding types (SURV, CPU, MEM, GC, THREAD, LATENCY, QUEUE). Max safe RPS, predicted memory/req, queue saturation warnings.

ml — Logistic Regression Engine sutra-ml

Why: Deterministic thresholds on individual features are brittle. Logistic regression learns the optimal weighted combination of all 14 signals. Pure Rust — no external ML framework required.

Logic

P(risky) = σ(wᵀ · x_scaled + b). Features z-score standardized. SGD + L2 regularization + learning rate decay. AUC via trapezoidal ROC.

Output

Risk probability per file, AUC score, confusion matrix. Serialized model (JSON weights, bias, means, stds).

llm — Validation Engine sutra-llm

Why: Static analysis produces false positives. An LLM provides contextual judgment — it evaluates whether a finding is a genuine production hazard. Disabled by default; opt-in enrichment layer.

Logic

Each finding sent to Ollama /api/generate. Prompt requests structured JSON: isValid, confidence, explanation. Response parser strips code fences and falls back to defaults.

Output

ValidationResult per finding. analyze_with_llm() runs deterministic engines first, then validates combined findings.

hitl — Human-in-the-Loop Engine sutra-hitl

Why: No automated analysis is perfect. The HITL engine closes the feedback loop — recording developer judgments and auto-adjusting future results. Over time, Sutra learns which engines produce reliable findings.

Logic

Feedback stored via FeedbackStore trait. For findings with ≥3 entries: confirm_ratio ≥ 0.8 auto-validates; reject_ratio ≥ 0.6 downgrades severity. Per-engine precision tracked.

Output

HITL-001 (precision), HITL-002 (coverage), HITL-003 (low precision) findings. Adjusted findings from adjust_findings().

orchestrator — Integration Layer sutra-orchestrator sutra-ci

Why: Individual engines produce fragmented signals. The orchestrator combines them into one answer: "Should I merge this PR?" SARIF and markdown output make it CI-native.

Merge Semantics

  • Risk: total_risk = max(engine.risk)
  • Findings: union across engines, deduped by ID
  • Metrics: per-field max()
  • Failures: engine errors produce ORCH-{ENGINE}-ERR findings

Output

Unified AnalysisResult, SARIF 2.1.0 JSON, markdown PR comments with emoji severity badges.

Design Principles

The architectural decisions that shape Sutra's development.

Math-first, ML-optional

Core risk score is deterministic. ML and LLM enrich, never replace foundational analysis. ML engine is self-contained with zero external framework dependencies.

Composable via traits

Each engine implements AnalysisEngine. Run independently, in batches, or in CI/CD. New engines implement exactly one trait with one function.

Deterministic by default

Same repo + commit = identical results (ML/LLM excluded). No hidden state, no non-determinism in the core pipeline.

Zero external runtime deps

Core stack requires only Rust, git, and a filesystem. No databases, queues, or containers required.

Quick Start

# Clone and build
git clone https://github.com/darshanredkar/sutra
cd sutra
cargo build --release

# Analyze with all engines
cargo run -- analyze /path/to/repo

# Single engine
cargo run -- analyze /path/to/repo --engine mgtg

# HTTP API server
cargo run -- server --port 8080

# Analyze via API
curl -X POST http://localhost:8080/v1/analyze \
  -H 'Content-Type: application/json' \
  -d '{"repo_path":"/path/to/repo","commit_hash":"HEAD"}'

# LLM validation (requires Ollama)
cargo run -- analyze /path/to/repo --llm

# SARIF for CI
cargo run -- analyze /path/to/repo --format sarif --output results.sarif
