End-to-End Failure Prediction Example

A comprehensive 1,040-line Python script that simulates a microservice repository on the brink of a production incident and walks through every engine, every formula, and every score computation in the Sutra pipeline.

The Synthetic Repo

  • 2 functions with cyclomatic complexity > 50 (untestable)
  • A circular dependency chain (processor → utils → processor)
  • A file with 45 revisions, 12 bug fixes, entropy 4.2
  • Tight co-change coupling between core modules
  • ML model predicts 94% defect probability
  • HITL feedback shows mixed precision across engines

What It Demonstrates

  • Cyclomatic & cognitive complexity (MGTG engine)
  • Tarjan SCC cycle detection (Dependency engine)
  • Shannon entropy & 14 JIT features (Process engine)
  • SGD logistic regression with AUC (ML engine)
  • LLM finding validation (LLM engine)
  • Feedback-driven auto-adjustment (HITL engine)

How to Run

python3 docs/example.py
Download example.py

Section Overview

The script is organized into 12 sections, each building on the last.

Section 1 — Foundational Math Utilities

sigmoid(z), shannon_entropy(), standard_scale(), weighted_sigmoid_score() — the math primitives shared by all engines.

Section 2 — Data Structures

AnalysisResult, EngineFinding, RiskLevel, FeedbackEntry — the types that flow through every engine, mirroring the Rust types in sutra-schema.

Section 3 — Synthetic Repository Generator

Builds a 15-file, 100-commit microservice repo with deliberate risk patterns: high-churn files, bug-dense modules, and files modified by many authors.

Section 4 — MGTG Engine

Cyclomatic complexity M = E − N + 2P per function. The walker identifies two functions exceeding 50 — well past the industry threshold of 10.

Section 5 — Dependency Engine

Tarjan's SCC algorithm detects a circular dependency. Fan-in and fan-out metrics identify the most coupled modules.

Section 6 — Process Engine

Shannon entropy per file, 14 JIT features (added_lines, deleted_lines, distinct_authors, bug_fixes, etc.), and co-change matrices that surface hidden coupling.

Section 7 — ML Engine

SGD + L2 regularization to learn optimal risk weights. Computes AUC via trapezoidal ROC, predicts 94% defect probability for the highest-risk file.

Section 8 — LLM Engine

Simulates Ollama validation of findings: each finding is sent to an LLM which returns isValid, confidence, and explanation.

Section 9 — HITL Engine

Stores feedback, computes per-engine precision, auto-adjusts findings based on majority voting with configurable thresholds.

Section 10 — Orchestrator

Runs every engine in sequence, fuses results using max-merge semantics, and prints a unified risk score and all findings.

Section 11 — Expected Output

Complete annotated output showing the risk score, all findings grouped by engine, and the 5 key insights Sutra produces about this synthetic repository.

Section 12 — Key Insights

Five actionable conclusions Sutra derives from fusing all 7 engine signals — demonstrating how multi-dimensional analysis catches what single-tool approaches miss.
